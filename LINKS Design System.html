The LINKS
Design System

Foundations and components that ship every LINKS portal. Calm, school-warm surfaces; weight-driven type; a single violet primary that ties the student experience together; and a portal-role color set that lets us flex into Instructor, Admin, Parent & Services without rewriting the chrome.

v1.0 · Student portal baseline styles.css · single source of truth Designed to grow with the product

Brand

The mark is a single rounded glyph in our primary violet, paired with the wordmark in Poppins. The wordmark is always paired with a portal sub-label (STUDENT PORTAL, INSTRUCTOR PORTAL, etc.) set in tight uppercase.

Typography

Two families: Poppins for display headlines and the wordmark, Inter for everything else. Hierarchy is established by weight contrast and tight tracking on display sizes — never by color alone, and never with more than three sizes per screen.

Display · H1 Welcome back, Alex. Poppins 600 · 28px / 1.15 / -0.02em
H2 / Section This week's classes Poppins 600 · 20px / 1.2 / -0.01em
H3 / Card title Algebra II — Period 3 Inter 600 · 15px / 1.4
Eyebrow DUE THIS WEEK Inter 600 · 11px / 0.08em uppercase
Body Open the assignment to read the prompt and start your draft. Your work autosaves every 30 seconds, and your teacher can see your progress. Inter 400 · 14px / 1.55
Caption Last edited 3 min ago Inter 400 · 12px / stone
Numerals · tabular 12 / 18 · 67% font-variant-numeric: tabular-nums

Type rules

Do
  • Pair Poppins display with Inter body — no other families.
  • Use weight (500 / 600 / 700) for hierarchy first; size second.
  • Apply tabular-nums on counts, scores, times.
  • Keep line-length under 72ch for paragraphs.
Don't
  • Don't use ALL CAPS for emphasis — use the eyebrow style for short labels only.
  • Don't introduce a third typeface (no Roboto, no system-ui fallbacks in chrome).
  • Don't use color as the only hierarchy signal — students with low vision must still see structure.
  • Don't go below 12px for any UI text.

Spacing

A 4-px base grid, with most components landing on 4 / 8 / 12 / 16 / 20 / 24 / 32. Card padding is 20–24px; vertical gap inside a card is 8–12px; gap between cards is 16–24px. A density toggle (data-density="compact") tightens row height and card padding for instructor and admin portals.

Motion

Restrained, school-Chromebook-friendly. Standard easing is cubic-bezier(0.4, 0, 0.2, 1); durations 120 / 160 / 200 / 320 ms. Fades and small (4–8px) translates only — no bounces, no spring physics, no Lottie.

120ms · hover

Background tint changes, icon color shifts. Snappy enough that hover feels reactive.

200ms · expand

Sidebar branch open/close, accordion, dropdown reveal. Pairs with chevron rotate.

320ms · drawer

Right-side drawers (AI Coach, message thread). Translate-X with eased ease-out.

Tactile elements

A small layer of scumorphic objects — tape, pins, polaroids, paper-cutout stamps — that sit on top of the clean foundation to add warmth and personality. Use them only at moments of celebration or human voice: achievements, streaks, pinned announcements, profile photos. Never on data, forms, dense lists, or anything a student needs to read fast. Rotation budget: ±3° on any single object — small enough that screenshots and screen readers still read it as straight.

Tactile objects sit on the regular cool-lavender surface — the warm-cream paper makes them read as a different material. Mix tape and pins; vary tilt by ±2°; never align them on a strict grid.
Rules
  • Use sparingly. Tactile objects are seasoning. A dashboard with one polaroid and one stamp lands; one with five lands as a craft project.
  • Reserved for celebration and personality. Achievements, streaks, profile photos, pinned notes, milestone moments. Not for data, forms, tables, calendar grids, navigation, or anything time-sensitive.
  • Rotation budget: ±3° max. Anything more makes screenshots look broken and copies less readable.
  • Warm paper on cool surface. The whole effect depends on the contrast between --paper-warm objects and the cool --surface desk. Don't put warm paper on a warm background.
  • Respect reduced motion. No tilt animations on hover; objects sit still.
  • Accessible labels. A stamp is decoration around real text — the achievement title and date must still live in semantic markup, not in the stamp's image.

Calendar

A single calendar component with four resolutions — Year, Month, Week, Day — and per-person filtering. The year view encodes event density as a heatmap (the more events on a day, the deeper the violet); clicking any day drills into the Day view. Person filters and the view switcher work together: filtering re-derives the heatmap and stack from only the visible people's events.

—
0 events shown Density 5+ events

How density is calculated

Per visible person

The heatmap reflects only events from people whose chip is active. Toggling a person off recolors the year instantly.

Five density tiers

0 / 1 / 2 / 3–4 / 5+ events. Tiers map to --bone → --student-100 → --student-300 → --student → --student-deep.

Click drills in

Clicking any year-view day jumps directly into the Day view for that date. Month-view cells do the same.

Iconography

LINKS uses Lucide — a clean, open-source line icon set already used in the portal. Stick to a single weight (1.6px stroke), use the slate token for default state, the student-deep token for active/selected, and never tint icons with the same color as text behind them.

Empty states

An empty state is the first impression of a feature. Use a feature-color icon, one short headline, one supporting line, and one primary action. Never show a blank panel — never use stock illustration filler.

The four kinds

No assignments yet

Your instructor will post work here. We'll let you know the moment it arrives.

Nothing matches "kepler"

Try fewer keywords or check spelling. We searched assignments, lessons, and messages.

Couldn't load your schedule

The connection dropped. Your work is safe — let's try again.

You're all caught up

No pending tasks for today. New work appears here as instructors post it.

Anatomy

Headline

State the situation in human language. "No assignments yet" not "Empty result set."

Supporting line

Explain what happens next or what to do. Keep under 18 words. Always reassuring, never apologetic.

Action

Exactly one primary CTA. If there's nothing actionable, omit the button — don't pad with "Learn more."

Loading & skeletons

Loading should feel like the page is already there. Use skeletons that mirror final layout for any wait > 200ms; spinners only inside button-bound async actions; progress bars for known-duration uploads/exports. Never block the whole screen for routine loads.

Rules

Match the shape

A skeleton's blocks should occupy the same regions as the final content — same circle for avatar, same row count, same widths.

Hold for ≥ 400ms

Don't flash the skeleton if real content arrives in < 200ms. Showing then yanking it feels worse than a brief blank.

Never spinner-blanket

A centered spinner over a whole panel is a signal of poor architecture. Render shell + skeletons; load regions independently.

Accessibility

A11y is non-negotiable in an education product — students and parents may use screen readers, keyboard navigation, low vision, or have vestibular sensitivities. Target WCAG 2.2 AA minimum; aim for AAA on critical text. Test with keyboard only and a screen reader at least once per release.

Focus & keyboard

Visible focus ring

:focus-visible ring is 2px var(--student) at 2px offset. Never outline: none.

Tab order matches reading order

Don't tabindex elements out of source order. Skip-to-main link hidden until focused.

Keyboard shortcuts

Modals close on Esc. Menus: ↑↓ to navigate, Enter to select. Calendar: ←→ moves the cursor by one unit of the active view.

Motion & reduced motion

Respect prefers-reduced-motion
/* Disable non-essential motion */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

Vestibular sensitivities can trigger nausea from parallax + auto-play. Always honor this preference.

Motion principles
  • Animations under 240ms feel responsive; over 400ms feel sluggish.
  • No autoplay video, no infinite carousels, no parallax on scroll.
  • Toasts and tooltips fade — they don't slide across the viewport.
  • Loading skeletons shimmer subtly (1.4s loop, low contrast).

Semantics & ARIA

Use real elements first

<button> not <div onclick>. <a href> for navigation. Form inputs in <label>.

Landmarks

Every page: <header>, <nav>, <main>, <footer>. Single H1 per page.

Live regions

Toasts and alerts use role="status" (polite) or role="alert" (assertive) so screen readers announce them.

Touch targets

Minimum hit target is 44×44 px (WCAG 2.5.5 AAA, also Apple HIG). Visually smaller controls (icon buttons, chips) get padding to reach this; the visible mark can stay smaller.

Page layout patterns

Every page in the portal follows one of three patterns: Dashboard (stats row + content grid), List view (filters + table or card list), or Detail view (page header + tabs + content). Sidebar + topbar chrome is shared.

L LINKS

The three page archetypes

Dashboard

Page header → stat row (3–4 cards) → primary content grid (2 col) → secondary feed.

Used for: Home, class overviews, my-time hub.

List view

Page header → filter bar → table or card list → pagination footer.

Used for: Materials, Messages list, Calendar table view.

Detail view

Breadcrumbs → page header → tab strip → tab content (often 2-col with sidecar).

Used for: Class detail, lesson, message thread, quiz.

Growing the system

This system is the v1.0 baseline — captured from the Student Portal as it stands today. As LINKS expands into the other six portals and into new product surfaces, follow these principles so the system grows without fracturing.

Adding a new component
  1. Compose from existing tokens first; never introduce a one-off color or radius.
  2. If a token doesn't exist, propose it as --name in styles.css with a semantic name (purpose, not appearance).
  3. Document it here with a demo card and a token row.
  4. Replace any inline values across the codebase with the new token.
Adding a new portal
  1. Reuse the same chrome (sidebar / topbar / page header).
  2. Swap the primary --student binding to that portal's role color via a body-level data attribute.
  3. Update the portal sub-label under the wordmark.
  4. Adjust nav labels and content; do not introduce new component patterns just for that portal.
When you can't find a fit

If a brief truly demands a pattern this system doesn't cover (a graph, a calendar grid, a media player), build it as a composition of tokens — borrow the card surface, the radii, the type scale — rather than treating it as license to invent new visual language.

Versioning

Tag the styles.css file with a version comment and bump on every breaking change. Document rename / removal in a changelog at the top of styles.css — consumers depend on the variable names.

One source of truth

This document is the contract between design and engineering. styles.css is the implementation. If they disagree, fix both — never let one drift.
